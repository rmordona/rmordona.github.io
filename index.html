Database Workload

Database systems - Oracle RDBMS in particular - offers an automated way to sample data every interval. In terms of dataset for this project, we are going to use one of the many database systems running at Oracle. For purposes of privacy and security, we are limiting data only around the generic database activities such as IO workload in terms of number of physical reads and number of physical writes, number of time an activity is in wait state, number time an activity spent time in the database, and number of database sessions.

Our goal is to be able to show how we review and analyze database workload. One way is to identify a change in database workload and thus be able to predict if the change in workload is something to be concerned about or not. The term "concerned" may be a loose term; but if there is such a case to be concerned about, decisions tend to be discussed around capacity planning wherein the topic would be whether to purchase more machines or more storage equipments to accommodate for growing workload.

To clarify what we have in the dataset, we will be going through 4 metrics (slides):

IO Workload: The dataset includes IO workload in the form of physical writes and physical reads. For a user transaction that requires to save a record of a bank deposit, we are measuring the act of writing a record of transaction to the database; in this respect, user transaction can be translated as one or multiple physical writes to the database. Similarly, for a user transaction that requires to retrieve bank statements, the transaction can be translated into one or multiple physical reads from the database.

User Activities: Underlying a user transaction are many individual computer tasks. A computer task may involve running programmatic code written in JAVA language or PLSQL language ( Procedural language for SQL) or merely just an inquery against the database (SQL). Here, a user activity is represented as a call to a program code. Without being specific to the functionality of a program code, we measure how many times a call has been made against a program code; basically, we measure how many times a JAVA code has been executed. In this context, we are looking at 4 categorical variables: JAVA execution, PLSQL execution, SQL execution, and all other combined calls ( termed as 'User Calls').

DB Time Spent: It helps to understand if transactions are taking too long to process. A transaction taking too long to process could be in a waiting state inside the database. We measure the time it takes for transactions to be in waiting state in the database. This is the DB time spent.

DB Sessions: A user that needs to transact requires to establish a user session to the database. We measure the number of user sessions that runs in the database. The higher the number of sessions, the higher the number of users; and hence, the busier the database system becomes.

IO Workload

The database system in use runs on a 12 powerful servers ( not to be specific on hardware model or type ). One of the first things to understand is to know if workload is equally distributed across the 12 servers. Because of wear-and-tear, we want to avoid an imbalance of load in that one side of a set of machines are very busy and the rest are idle. Machines that are more busy may consequently suffer hardware failures. Measuring a balanced IO workload, it can be observed from the chart below server 1 seems to accommodate more physical reads than the other servers. It would be good to understand why this is the case - a decision would then be to work with different teams to investigate the cause.

DB Time Spent

Life experiences have taught us that whenever we are waiting in line, whether in a bus station or train station, it could mean that there is alot of waiters waiting or there is not enough bus or train for everybody. Similarly, every time a transaction is waiting, it could be that there is not enough resource ( physical machine ) to process incoming transactions; otherwise, there are just too many transactions that suddenly happened. Here we have chosen 4 common activities that are observed to be in wait state. Those common activities are mapped correspondingly to database wait times and there are 4 of them: JAVA execution, PLSQL execution, SQL execution and then all other 'DB time'. In the chart, it can be observed that SQL execution tends to have the most number of waits compared to db time for JAVA and PLSQL which tell us that users seem to be more busy querying information and taking longer than updating records. On the other hand, the 'DB time' shows higher waits in the month of May. The goal here is to reduce wait times. The lesser the waits, the more efficient the database system is. To be able to determine why month of May has more DB time spent, there are available tools that will help to investigate this further and be able to narrow down the cause to individual queries or to hardware resource issues.

Summary and Disclaimer

Summary: We have shown four different metrics of analyzing and understanding database workload. In real-world, there are many more metrics that need to be accounted for. Each of those metrics could show signs of change or anomaly and it is the responsibility of technical support to be able to review and analyze those signs and work with other groups to address them. At the end of the day, the goal is to prevent systems from breaking, be able to plan for sufficient capacity, and be able to improve customer experience.
