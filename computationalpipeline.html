The Power and Art of Approximation

Chapter 14 Distributed Computation

The end result of all our efforts by going through data processing and modeling is a well-trained ML model. The next question to ask is how to manage this ML model. So far we have not covered the concept of Model Management in great detail and it is crucial that models go through some automated, distributed, and collaborative life-cycle management. While this topic requires an understanding of technology, we try to avoid highlighting any specific technology. Instead, let us focus on what is available in R. We know that technology comes and goes. So the takeaway in this chapter is to build some foundation on how to manage models.

14.1 Integration and Interoperability

ML models are used mostly for inferencing. Our goal is to use ML models to predict given new data. However, there may be cases when one model may need to be shared across frameworks and platforms. This is where we need to consider Portability of models. In this section, we shall introduce the idea of Portability to address Integration and Interoperability of ML systems.

In any practical sense, we normally rely on ML frameworks and ML libraries instead of building our own ML algorithms from scratch. However, as we begin to work in more complex systems and bigger organizations, we begin to see three challenges. The first challenge is that not all organizations are locked into only one framework or to only one programming language. And so while there are too many ML frameworks and ML libraries available across a variety of programming languages, it is crucial to recognize that when using any specific framework, the generated ML model results in a format that is different from one that is generated by another framework. The second challenge is that while certain frameworks may serialize the format into a common format such as XML and JSON, the structure or schema layout may be different. Each framework would have to agree upon a common structure or a schema layout in what we call a contract that is accompanied by rules and restrictions. The third challenge is that scoring a prediction in the context of ML models may rely upon the data and how the data is trained - the key is the set of operators that will help a scoring engine to perform correct inference. It may be easy for a simple linear regression model to perform certain fundamental statistical and mathematical functions - such functions can be operationalized into the schema. Such basic operations can be implemented by different frameworks. However, for models trained by methods that are layered and nested such as Deep Learning, then we have to be able to allow extensibility.

Therefore, to be able to develop an ML application relying on any framework that can interoperate with others, the three solutions need to be in place, namely, having a common format, having a common schema specification, having a common operation with the ability to accommodate other operations. In this chapter, we shall introduce three Open Standards, namely PMML, PFA, and ONNX.

Note that we do not cover data serialization systems in this book such as Avro and ProtoBuf. We encourage readers to investigate the two systems and their specifications however.

In terms of integration, this is where Pipelines and API end-points are needed. The discussion of the concept follows next.

14.2 ML Pipelines

One would wonder how members of a Data Science team can work together in an environment in which there is an ecosystem of many moving parts and many stakeholders. One option is perhaps to rely on the concept of ML Pipelines. Model Pipelining is the method of identifying atomic units of work and organizing them into stages, then allow an input to be pipelined from one stage to the next in an automated manner. While there may be variations of an ML Pipeline in terms of the modules and stages, let us stick to the generic variant which comprises of four main stages, namely Cleaning, Preprocessing, Training, and Evaluation. In the figure, we see an abstract of modularized pipelines. If we allow flexibility, in a microservice fashion perhaps in which each module may be a microservice, we can pipeline the input from stage 1 module 1, stage 2 module 6, stage 3 module 10, and stage 4 modules 12 and 14. For example:

While pipelines are composed of many units of work that are stitched in different ways, such abstraction can be implemented through the use of Big Data. Obviously, it takes a team to design and architect the hardware ( CPU, memory, storage, network) and software to make this possible. Deployment of such an environment depends in part upon the nature and volume of data. Otherwise, such resources are already made available in cloud-based service offerings that can possibly handle a massive amount of data.

14.3 Open Standards

The idea of having an open standard format is to allow models to be portable and sharable. To do that, the format produced by each algorithm implemented by each framework needs to be converted into a common standard format. We need converters that support a wide range of ML algorithms for every framework we use to become part of a common standard. Once the formats are standardized, we then need an engine to be able to consume the standardized models inorder to perform scoring and inferencing. This is where we use the terms producers and consumers. For example, certain standards are developed along with a runtime to be able to consume the model and perform the inference. We shall come to this concept when we talk about PFA and ONNX.

Note that in our Machine Learning chapters, we have demonstrated the use of saveRDS(.) and readRDS(.) functions to preserve our models so that we can reuse them as we need without having to go through the training process. However, the two functions do store our model in Rdata format and thus may not be portable across any other platforms. Here, we shall now illustrate how our Rdata can be transformed into a standard format that we can use to transport our models.

Note that the invalidValueTreatment attribute is left empty to fit the text in this book. Nonetheless, consumers of the PMML-based model can use the MiningSchema to evaluate fields in the schema for validity. For example, if values are invalid or missing, or if the data type is incorrect, then the application can prescribe remedies (or treatments).

Now, it is essential to point out that PMML-based model producers have to strictly follow the PMML standard to be truly portable while at the same in order to avoid unnecessary efforts for the consumers of the schema to patch additional workarounds. But even then, a Document Type definition (DTD) for PMML can be downloaded and used by consumers of the model schema to validate that PMML rules are correctly followed.

Note that the PMML specification includes a DataDictionary structure, MiningField structure, an Output structure, RegressionTable structure, and the RegressionModel structure. The RegressionModel describes the algorithm used (which is Least-Squares in this case). The RegresssionTable structure keeps a record of the model’s coefficients to be used for inference. The DataDictionary and MiningField structures describes the dataset used by the model along with the data type (e.g. continuous and double).

While not seen in our example, note also that PMML specification includes data transformations. As an example, such XML-based transformations include normalization, discretization, aggregation, and other transformation functions. However, PMML is not extensible enough as pointed out by Jim Pivarski in his PFA paper as it does not allow accommodating other functions or operations. Let us discuss PFA next.

14.3.3 Open Neural Network Exchange (ONNX)

Another open standard format is ONNX made available as an open-source by Facebook and Microsoft which supports standard format for deep learning models. The standard format follows the concept of acyclic graph in which nodes represent the operators and edges represent the tensors. These tensors can be treated as messages that are passed from node to node (as we have shown in CNN). It is natural therefore to see tools such as Netron that can graphically present ONNX models.

Note that while PFA follows Avro specification for data structures, ONNX follows Protocol Buffer specification.

ONNX may continue to receive wide acceptance and may continue to evolve in terms of supporting other frameworks. At this point, we encourage readers to investigate any available support for frameworks in R. As a starting point, one can investigate onnx R package (Yuan T. 2021).

library(onnx)

In terms of inference, a scoring engine called ONNX runtime is available. Any ONNX model can be ingested into the runtime to perform prediction. It helps to investigate further the devices that support the ONNX runtime. As it stands, one of the goals is to be able to deploy the ONNX runtime for Edge computing to extend the ability of Edge devices to perform distributed ML computing, at least for inferencing.
